Fix xargs ARG_MAX issue in parallel_runner.sh for large task lists

Fixed "command line cannot be assembled, too long" error when running marker
agents with many tasks (e.g., 224 tasks = 7 activities × 32 students).

## Problem

When processing structured assignments with many students and activities, the
parallel_runner.sh script failed with:

```
xargs: command line cannot be assembled, too long
```

This occurred because xargs was trying to pass very long command lines (with
paths containing spaces) as arguments, exceeding the system's ARG_MAX limit.

## Root Cause

The original xargs implementation used `-I {}` to substitute entire command
lines into the bash command template:

```bash
cat "$TASKS_FILE" | xargs -I {} bash -c "execute_task \"{}\" ..."
```

For 224 tasks, each containing paths like:
```
python3 '/Volumes/Mac Storage/.../file.py' --arg '/Volumes/Mac Storage/.../file.ipynb' ...
```

The assembled command line exceeded ARG_MAX (typically 256KB on macOS).

## Solution

Instead of passing command lines through xargs arguments, the new approach:

1. **Passes line numbers** (1, 2, 3, ... N) through xargs
2. **Reads commands from file** using `sed -n "${line_num}p"` inside each worker
3. **Avoids ARG_MAX** entirely since line numbers are small integers

**Before:**
```bash
cat tasks.txt | xargs -I {} bash -c "execute_task \"{}\" ..."
# Passes long commands through xargs ’ ARG_MAX error
```

**After:**
```bash
seq 1 N | xargs -I {} bash -c 'execute_task_by_line "{}" "tasks.txt" ...'
# Passes only line numbers through xargs ’ no ARG_MAX issues
```

## Implementation

**src/parallel_runner.sh:**

- Added `execute_task_by_line()` function that:
  - Takes line number and tasks file path
  - Uses `sed -n "${line_num}p"` to read specific line
  - Calls original `execute_task()` with the command

- Modified xargs execution to:
  - Generate line numbers with `seq 1 $TOTAL_TASKS`
  - Pass numbers through xargs instead of command lines
  - Call wrapper function that reads from file

**Benefits:**
- Handles arbitrarily large task lists (thousands of tasks)
- Works with paths containing spaces and special characters
- No ARG_MAX limitations
- Maintains parallel execution capabilities
- Same performance as before

## Compatibility

- Works with both GNU xargs and BSD xargs (macOS)
- Uses standard `seq` and `sed` commands (available on all Unix systems)
- Falls back gracefully to sequential execution if xargs unavailable
- GNU parallel path remains unchanged (preferred when available)

## Impact

This fix enables the marking workflow to process large assignments with hundreds
of submissions without hitting command-line length limits. The workflow can now:

3. ’ Continue to Normalizer stage

The fix is transparent to users and requires no configuration changes.
